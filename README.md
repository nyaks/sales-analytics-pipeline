# Sales Analytics Pipeline

A complete end-to-end data engineering pipeline that generates synthetic sales data, runs an ETL process, and feeds a live Streamlit analytics dashboard with forecasting capabilities.

---

## Quick Start

```bash
# 1. Create virtual environment
python -m venv venv && source venv/bin/activate
# On Windows:  venv\Scripts\activate

# 2. Install dependencies
pip install -r requirements.txt

# 3. Generate synthetic data (~12,000 rows)
python generate_sales_data.py

# 4. Run the ETL pipeline (CSV → transformations → SQLite)
python etl_pipeline.py

# 5. Launch the dashboard
streamlit run dashboard.py
```

Open `http://localhost:8501` in your browser.

## ETL Pipeline Details

The pipeline is implemented in `etl_pipeline.py` with three clearly separated stages:

### Extract
- Reads the raw CSV file (`sales_raw.csv`) generated by `generate_sales_data.py`
- Loads 12,000 rows into a pandas DataFrame

### Transform
Seven transformation steps:
1. **Data Cleaning** — Median imputation for missing `unit_price` (per category), mode imputation for missing `product_name`, and revenue recalculation from `quantity × unit_price` where revenue was null
2. **Customer Lifetime Value** — Aggregates total revenue, order count, average order value, and tenure (days) per customer
3. **Monthly Cohort Analysis** — Assigns each customer to a cohort based on their first purchase month, builds a retention matrix showing customer activity rates across subsequent months
4. **Product Performance** — Revenue, quantity, and order metrics per product and per category
5. **Regional Breakdown** — Revenue, order count, and unique customer stats per region
6. **Monthly Revenue Summary** — Revenue by month with calculated Month-over-Month (MoM) growth percentages
7. **Salesperson Leaderboard** — Ranked by total revenue with supporting metrics

### Load
- Each transformed DataFrame is written as a separate table in a SQLite database (`sales_analytics.db`)
- 10 named tables, each queryable independently by the dashboard

---

## Forecasting

Two methods are available in the dashboard:
- **Linear Regression** — Fits a trend line on the indexed monthly revenue series, projects 3 months forward
- **Exponential Smoothing** — Simple exponential smoothing (SES) with configurable alpha for weighted historical averaging

## Interview Talking Points

### 1. ETL Architecture & Data Quality
> This project demonstrates a production-style ETL pipeline with clearly separated Extract, Transform, and Load layers. The cleaning step handles realistic data quality issues—missing prices, null revenues—using statistically sound imputation (median per category rather than global averages), which prevents skewing product pricing distributions. In production, this same pattern scales to Airflow DAGs with data quality gates at each stage.

### 2. Customer Lifetime Value (CLV) Modeling
> CLV is foundational for marketing ROI and customer segmentation. Here I calculate it as the sum of all historical revenue per customer, combined with tenure and order frequency metrics. This is the "historical CLV" methodology—distinct from predictive CLV that uses probability models (BG/NBD). The CLV computation uses groupby aggregations that would translate directly to SQL or Spark for billion-row scale datasets.

### 3. Cohort Analysis & Retention Matrices
> Monthly cohort analysis is a standard analytics pattern for SaaS and e-commerce. By assigning customers to cohorts based on their first purchase month and tracking activity rates over time, we can visualize whether retention is improving or declining—a leading indicator for product health. The heatmap representation makes cohort-age effects immediately visible to stakeholders.

### 4. Time-Series Forecasting
> The dashboard implements two complementary forecasting approaches: linear regression (captures trend, assumes steady growth/decline) and exponential smoothing (weights recent data more heavily). The ability to compare both methods in real-time helps validate forecast reasonableness. In production, I would extend this to SARIMA or Prophet for datasets with seasonality, and add confidence intervals.

### 5. Dashboard Engineering & Data Modeling
> The Streamlit dashboard connects to pre-computed summary tables in SQLite rather than querying raw orders. This "semantic layer" approach is critical for performance—instead of scanning 12,000 rows per interaction, the dashboard reads pre-aggregated tables with hundreds of rows. This mirrors the dimensional modeling pattern used in production data warehouses (Star Schema), where fact tables are joined and pre-computed into dimension-specific summary tables.

### 6. Scalability & Data Engineering Best Practices
> While this prototype uses pandas and SQLite for portability, the architecture is designed for horizontal scaling: the ETL transformations (groupby, merge, pivot) map directly to PySpark or DuckDB operations, the SQLite layer would be replaced by PostgreSQL or Snowflake, and the pipeline would be orchestrated by Airflow or Dagster with incremental loading (only processing new rows since last run). The separation of concerns also enables unit testing each transformation independently.
